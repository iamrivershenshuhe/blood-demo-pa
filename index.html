A note on craft
為什麼這個比「翻 SOP」好

傳統 SOP 文件寫得完整,但實際處置時同仁要自己判斷現在在哪個分支。系統把這件事翻過來——一次只問當下需要的條件,根據回答決定下一步。

每一步引用的章節與版本都會明確標示,既能讓新人快速上手,也方便事後稽核追溯。

A note on craft
為什麼要追溯到條文版本

法規會修,實務作業也會跟著變。系統的每個回答都標註法規的最新修正日期與條文位置,使用者一眼就知道這個答案依據的是哪個版本。

這對品保、稽核、面對主管機關查驗時都是必要的——不是「AI 說的」,是「依據哪一條哪一版」。

Demo iii — On knowing what one doesn't know

它什麼時候會說不知道

A good system fails loudly when out of scope. 對醫療場景特別重要——亂答可能造成真實的傷害。這個 demo 展示系統怎麼識別「我不該回答這個」,以及面對範圍外問題時的退讓設計。

Scope discipline — demo mode ready
— pick a question to see how it responds
A note on craft
退讓不是缺陷,是設計

一般 RAG 系統最大的危險不是不知道答案——是不知道自己不知道,然後編一個。對血液業務來說這比沒答案還可怕。

本系統的設計原則:沒有對應的知識庫片段就明確退讓,並建議使用者去哪裡找正確答案。Citations 永遠是真實的,範圍外問題的 citations 一定是空的——這個機制讓使用者可以信任「有引用就是有依據」。

觸發症狀:頭暈、噁心、面色蒼白、冷汗、心跳加速或減緩。

即刻處置:

  • 停止採血,保留針頭備用
  • 協助捐血者平躺,雙腿抬高 15–30 度
  • 鬆開頸部、腰部緊束衣物
  • 密切觀察生命徵象,每 5 分鐘記錄一次
  • 若意識清醒,可少量補充溫水或含糖飲料

持續性反應(超過 15 分鐘):通報醫師到場評估,必要時送醫或叫救護車。完整紀錄於良心回電系統,並追蹤捐血者後續狀況。

點上方按鈕觀察系統反應
A note on craft
為什麼保護要做在三個層面

視覺浮水印確保截圖後仍能追溯到使用者與時段;互動攔截讓常見的「順手複製貼到 LINE」失效;稽核紀錄則讓所有異常操作可被事後追查。

實際部署版本還包含螢幕擷取偵測(macOS / iOS)、定期重新驗證 session、以及異常行為自動告警。這些都不是要「鎖死」使用者——而是讓「敏感內容外流」這件事變得有摩擦力、有紀錄、可追蹤。

Demo v — System architecture

系統架構

Five layers, each with its own responsibility. 從使用者介面到資料層,點開每一層看它做什麼、用什麼技術、為什麼這樣切分。

A note on craft
為什麼要分這麼多層

把職責切清楚,後續才能各自獨立替換——例如把雲端 LLM 換成地端模型,只動「生成層」;新增一份法規,只動「資料層」與「檢索層」的索引。每層之間靠明確的契約溝通,不互相耦合。

這也是為什麼 RAG 系統不是「接個 ChatGPT API」這麼簡單——每一層都有自己的設計取捨,加總起來才是一個能上線給同仁日常使用的系統。

Demo vii — A primer

RAG 是什麼

A short, jargon-free explanation for anyone in the room. 不需要懂技術,五分鐘看完就理解這套系統的本質。

i/

一般人想像中 AI 怎麼回答問題

It's like asking a brilliant generalist who's read everything once.

大型語言模型(ChatGPT、Claude)被訓練過大量網路資料,有點像找了一個讀過很多書的人。你問問題,他憑記憶回答。

但這個「記憶」是模糊的——它記得大概,記不得精確;它讀過公開資料,沒讀過你公司的內部文件。

你問
捐血者血色素的標準是?
它答
「男性大概 13、女性大概 12」 — 接近但無法給出依據,也可能版本過時
ii/

這個方式的問題

It might be confident. It might be right. It can't show you where it got that.

對閒聊或寫詩沒差,但對醫療業務這是致命缺陷:

  • · 沒辦法引用法規條文與版本
  • · 不知道你血基會內部的 SOP
  • · 可能「幻覺」——很有信心地答錯
  • · 資料傳到 OpenAI / Anthropic,違反個資要求
iii/

RAG 的解法

Instead of asking from memory — look it up first, then answer.

RAG(Retrieval-Augmented Generation)的精神很簡單:先檢索,再回答。

像一個圖書館管理員,他不憑記憶回答你——他先找出相關書籍、翻到相關段落,然後對著書本告訴你答案。每句話都有依據,每個依據都可以翻回原書查證。

iv/

RAG 的三步驟

Three steps. Boringly simple, deeply effective.

1 · Retrieve
從文件庫中找出與問題最相關的幾段文字
2 · Augment
把這些片段塞給 LLM 當作回答時的依據
3 · Generate
LLM 根據這些片段組織出答案,並標註引用來源
v/

為什麼適合血基會

Because the corpus is specific, the answers must be auditable, and the data must stay inside.

血基會的知識(SOP、捐血須知、健康標準、法規)有三個特性,正好對應 RAG 的優勢:

·
專屬內容——LLM 沒讀過你的內部 SOP,但 RAG 可以即時注入
·
須能追溯——每個答案都對應到具體條文/章節/版本
·
資料敏感——地端部署,文件不離開機房
In other words
為什麼這個技術是現在的選擇

RAG 不是新技術,但它在 2024–2025 年成熟到能上線給專業領域使用——主要是因為 Local LLM 變強了、檢索演算法變好了、向量資料庫變便宜了。三個條件同時到位,使得「資料不離開機房 + 引用可追溯 + 對話自然」第一次同時可能。

對血基會而言,現在剛好是切入點:技術已成熟,成本已下降,合規優勢明顯。

我們和一般 RAG 的差別

"RAG" is a hot word; many companies say they do it. Here are six places where the work is. 每一項都是一個具體設計選擇——一般做法 vs 我們做法的對照。前面 demo 看到的細節,在這裡收束成一個論述。

In closing
這六項加起來才是「適合血基會的 RAG」

每一項單獨看都不複雜,但 RAG 的價值來自六項都做對——只要有一項偷懶,整個系統的可靠度就會掉一個檔次。對醫療業務來說,系統的可靠度不是「夠用就好」,而是「不能出錯」。

這也是為什麼這套 spec 寫了 11 個 stages、超過 200 頁的設計文件——不是炫技,是把每個取捨講清楚,讓未來接手的人知道為什麼這樣設計。
